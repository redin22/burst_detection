# Burst Detection
Burst detection using CUSUM change-point detection and DBSCAN clustering

This project involves the implementation of 2 burst detection methods: CUSUM change-point detection on pressure time-series for areas with sensor coverage, and customer contact spatial/temporal clustering for areas without sensors. The final outputs are 2 alert feeds generated by each of the detection methods to show potential incidents, their severity, advisory radius, and the number of pipes affected. You will also get 2 validation results reports showing false positives, false negatives, precision and recall for each of these two methods against the provided validation set.

## Data Inputs

The notebook uses the following datasets:

- **Pressure readings:** timestamped pressure values by site.
- **Pressure sites:** site locations used to map readings geographically.
- **Pipes:** pipe geometries used to identify candidate affected pipes.
- **Customer complaints:** complaint type, priority, timestamp, and location.
- **Work orders:** historical work order records that may provide pipe IDs and addresses.

## Limitations and Shortcuts

This prototype makes several pragmatic choices to move quickly:

- The CUSUM method uses a rolling 24-hour baseline rather than a more robust time-of-day baseline across multiple days or weeks.
- DBSCAN parameters were estimated from complaint volume rather than fully tuned.
- Severity, location confidence, and advisory radius are rule-based and should be recalibrated with more domain knowledge.
- Pipe topology, valves, pressure zones, pipe age, pipe material, and asset health are not fully incorporated.
- Validation is limited to the sample confirmed events provided.

  ## Production Readiness Considerations

To make this prototype production-ready, I would focus on five areas:

1. **Real-time telemetry integration:** connect to SCADA or telemetry systems with freshness and data quality checks.
2. **Threshold calibration:** tune CUSUM, DBSCAN, severity, confidence, and advisory radius rules using confirmed incidents.
4. **Operational workflow:** add alert acknowledgement, assignment, escalation, resolution, and feedback capture.
5. **Reliability monitoring:** monitor ingestion, model runs, stale readings, alert delivery, and dashboard availability.
